## Фаза 1. Багфиксы карточки товара (быстро, сразу видно)

1. **Кликабельные коллекция и категория** в карточке товара
   - Сверху (где сейчас «BLU MEDITERRANEO») → ссылка на коллекцию `/collection/blu-mediterraneo`
   - Под названием (где «Тостеры · Pastel Blue») → «Тостеры» ссылкой на каталог с фильтром по категории, «Pastel Blue» — ссылкой на каталог с фильтром по цвету
   - Те же ссылки на превью-карточках в каталоге (по hover/клику только название уходит в товар, а служебные подписи — в каталог)

2. **Переключатель цвета**: текущий вариант сейчас исчезает из списка
   - Показываю всю «семью» (`model_group`), текущий цвет помечаю выделенной обводкой и не убираю из ряда
   - Чиню активное состояние и порядок (сортировка по `colour`)

## Фаза 2. Поиск (каталог + админка)

3. **Полнотекстовый поиск в каталоге**
   - Иконка-поиск в шапке сайта → выпадающая панель с подсказками (товары появляются по мере ввода)
   - Поиск по: имени, SKU, EAN, описанию, цвету, категории, коллекции — на трёх языках (когда они появятся в Фазе 4)
   - На уровне Postgres: добавляю `tsvector` колонку + GIN-индекс, обновляется триггером. Без сторонних сервисов.

4. **Поиск в админке**
   - Тот же `tsvector` + фильтры (категория, наличие, опубликовано/скрыто, новинка/хит)
   - Сортировки: по дате, по остатку, по цене

## Фаза 3. Остатки и срок доставки

5. **Новые поля у товара**: `stock_qty` (на складе), `stock_reserved` (в резерве), `lead_time_days` (срок доставки на склад в Армении)
6. **Витрина**:
   - есть на складе → «В наличии»
   - 0 на складе, есть `lead_time_days` → «Под заказ, ~14 дней»
   - 0 и срока нет → «По запросу» (как сейчас)
7. **Резерв при заказе**:
   - При создании заказа `stock_reserved += qty`
   - При статусе «выполнен» — списываем `stock_qty -= qty`, снимаем резерв
   - При статусе «отменён» — снимаем резерв
   - В каталоге показываем доступное = `stock_qty - stock_reserved`
8. **В админке**: колонка «Остаток / резерв / срок», массовое редактирование остатка, история изменений (минимально — лог)

## Фаза 4. Трёхязычность всего контента (RU/EN/HY)

9. **Схема в БД** — добавляю поля переводов:
   - `products`: `name_en`, `name_hy`, `description_en`, `description_hy`, `specs_en` (jsonb), `specs_hy` (jsonb)
   - `collections`: `name_en`, `name_hy`, `description_en`, `description_hy`
   - `themes`: `name_en`, `name_hy`, `description_en`, `description_hy`
   - `color_swatches`: `name_en`, `name_hy` (но «Blu Mediterraneo», «Dolce & Gabbana», «Coca-Cola», «Smeg500» оставляем как есть — это бренды)
   - Категории: создаю отдельную таблицу `categories` с переводами (сейчас это просто текстовое поле в `products`)
10. **AI-перевод оптом** через Lovable AI (gemini-3-flash):
    - Скрипт прогоняет всё, пишет в БД
    - Бренды/коллаборации оставляю в исходном виде по списку: Blu Mediterraneo, Dolce & Gabbana, Sicily Is My Love, Divina Cucina, Coca-Cola, Smeg500, Porsche, FAB28/FAB30/…
11. **Админка**:
    - На странице товара — вкладки RU / EN / HY для названия, описания, характеристик
    - Кнопка «Перевести через AI» на каждом поле — для новых товаров
12. **Сайт**: все компоненты читают поля по текущему языку с фолбэком на RU. Sitemap отдаёт hreflang.

## Технические детали

- БД: миграции по фазам, чтобы можно было остановиться между ними
- Поиск: `tsvector` + GIN, конфигурация `simple` (мультиязык), триггер обновления
- Перевод: серверный скрипт через Lovable AI Gateway, идемпотентный (можно перезапускать)
- i18n: расширяю существующий `src/lib/i18n.tsx` хелпером `tProduct(product, field)` который сам выбирает `field_<lang>` с фолбэком
