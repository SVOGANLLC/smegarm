## Фаза 5 — CRM, уведомления, бот, PWA

### A. CRM статусы и роли
- Статусы заказа: `new → confirmed → shipped → done` + `cancelled` (триггер резерва уже привязан к этим статусам).
- Триггер: резерв ставится сразу при создании. Списание со склада — на `done`. Освобождение — на `cancelled`.
- Роли в `user_roles`: добавить `manager` к существующим `admin`/`user`. RLS:
  - Менеджер видит и меняет заказы/запросы, добавляет внутренние заметки, не может управлять каталогом/ценами/пользователями.
  - Админ — всё.
- Новые поля заказа: `internal_notes` (text), `status_history` (jsonb лог переходов с автором и временем).
- Канбан-вид и таблица в `/admin/orders` с фильтром по статусу, поиском, экспортом.

### B. Экспорт заказов в Excel
- Серверная функция `exportOrders` (createServerFn) → xlsx (sheetjs или exceljs) → возврат base64 + скачивание из админки.
- Колонки: №, дата, статус, клиент, контакты, адрес, позиции (SKU/название/qty/цена), сумма, заметки.

### C. E-mail уведомления
- Канал: Lovable Emails (встроенная инфраструктура, без сторонних ключей).
- Адрес получателя для менеджеров: **smeg@smeg.am**.
- Шаблоны (RU/EN/HY, выбор по `lang` заказа/запроса):
  1. Менеджеру — «Новый заказ» (триггер: создание заказа).
  2. Менеджеру — «Новый запрос цены/контакта» (триггер: inquiry).
  3. Клиенту — «Заказ готов» (триггер: смена статуса на `done`).
- Реализация через `createServerFn` + очередь `email-templates`. Язык — поле `lang` на orders/inquiries (сохраняем текущий язык сайта в момент отправки).

### D. Telegram-бот
- Connector Telegram (workspace-level).
- Команды: `/start` (привязка чата к менеджеру по коду), `/orders` (последние), `/status <id>` (смена).
- Уведомления в указанный чат на: новый заказ, новый запрос, смена статуса, низкий остаток (`stock_qty - stock_reserved <= 1`).
- Webhook: `src/routes/api/public/telegram/webhook.ts`.

### E. PWA (установка на главный экран)
- Манифест + иконки + theme-color. Без service-worker (offline не запрашивали).
- Баннер «Добавить на главный экран» при втором посещении.

### F. ConverseBank — онлайн-платежи
- Уточнить у заказчика: есть ли merchant ID / тестовый стенд / документация. Это банк-специфичная интеграция, нужны кредлы.
- План: серверный endpoint `createPayment` → редирект на платёжную форму банка → webhook `/api/public/converse/callback` с проверкой подписи → меняем статус заказа на `confirmed`/`paid`.
- Когда заказчик пришлёт доступы — реализуем.
